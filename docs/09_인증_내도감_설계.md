# FishNote — 인증 + 내 도감 설계 (Phase 4, Codex용)

> 목표: Spring Security + JWT 학습(기획서 Phase 4) + 북마크의 서버 승격.
> 대원칙: **로그인은 선택.** 비로그인 사용자는 지금처럼 모든 기능(익명 후기, localStorage 저장)을 그대로 쓴다. 로그인하면 "기기가 바뀌어도 유지되는 내 도감"이 생길 뿐이다.
> 순서: A1(BE 인증) → A2(BE 북마크) → A3(BE 후기 연결) → A4(FE 인증 상태) → A5(FE 화면) → A6(FE 북마크 서버화+병합) → A7(FE 후기 폼) → A8(검수).

## 0. 설계 결정 (근거)

| 결정 | 이유 |
|---|---|
| 토큰: Bearer + localStorage | FE(fishnote.kr)·BE(onrender.com)가 크로스 도메인 → 쿠키는 Safari ITP 등에서 차단됨. 헤더 방식이 이 구조의 정석 |
| access 토큰 단일(7일), refresh 없음 | v1 학습 범위 조절. refresh 도입은 v2 확장 포인트로 문서화만 |
| 가입: 이메일+비밀번호+닉네임, 이메일 인증 없음 | v1 스코프 아웃(스팸 가입은 트래픽 생기면 대응) |
| 익명 후기 유지 | 후기 진입장벽을 올리지 않는다. 회원 후기는 "관리가 편해지는" 보너스 |
| 비밀번호 정책 | 8~64자, 그 외 제약 없음(과한 규칙은 이탈 유발). BCrypt 해시(기존 인코더 재사용) |

## 1. DB 확장

```sql
CREATE TABLE users (
    id            BIGSERIAL PRIMARY KEY,
    email         VARCHAR(255) NOT NULL UNIQUE,
    password_hash TEXT NOT NULL,
    nickname      VARCHAR(30) NOT NULL,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE user_bookmark (
    user_id    BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    fish_id    BIGINT NOT NULL REFERENCES fish(id) ON DELETE CASCADE,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    PRIMARY KEY (user_id, fish_id)
);

ALTER TABLE review ADD COLUMN user_id BIGINT REFERENCES users(id) ON DELETE SET NULL;
```

- `review.user_id`는 **nullable** — 익명 후기는 NULL 그대로.
- JPA: `User` 엔티티, `UserBookmark`(복합키 @IdClass 또는 @EmbeddedId — 학습 포인트), `Review`에 `@ManyToOne User user` 추가.

## 2. API 명세

### 인증 (`/api/v1/auth`)
| 메서드 | 경로 | 설명 |
|---|---|---|
| POST | `/auth/signup` | `{ email, password, nickname }` → 201 `{ id, email, nickname }` |
| POST | `/auth/login` | `{ email, password }` → 200 `{ accessToken, nickname }` |
| GET | `/auth/me` | Bearer 필요 → `{ id, email, nickname }` |

- 검증: email 형식·중복(중복 시 409 "이미 가입된 이메일이에요"), password 8~64자, nickname 1~30자.
- 로그인 실패는 이메일/비밀번호 어느 쪽이 틀렸는지 구분하지 않고 401 "이메일 또는 비밀번호를 확인해 주세요" (계정 존재 노출 방지).
- JWT: HS256, `JWT_SECRET` 환경변수(32바이트 이상), 만료 7일, subject=userId.

### 내 도감 (`/api/v1/me/bookmarks`, 전부 Bearer 필요)
| 메서드 | 경로 | 설명 |
|---|---|---|
| GET | `/me/bookmarks` | 내 북마크 생선 목록 (`FishSummaryResponse[]`, 저장한 순) |
| PUT | `/me/bookmarks/{fishId}` | 추가 (멱등, 이미 있으면 204) |
| DELETE | `/me/bookmarks/{fishId}` | 제거 (없어도 204) |
| POST | `/me/bookmarks/merge` | `{ fishIds: [1,3,5] }` — localStorage 병합용 일괄 추가, 존재하는 fish만 반영 |

### 후기 변경
- `POST /fish/{id}/reviews`: Authorization 헤더가 **있으면** user 연결 — nickname은 회원 닉네임 사용, password 불필요(요청에 없어도 통과). **없으면** 기존 익명 규칙 그대로.
- `DELETE /reviews/{id}`: 내 후기(`review.user_id == 토큰 userId`)면 password 없이 삭제. 익명 후기는 기존 비밀번호 방식.
- `ReviewResponse`에 `mine: boolean` 추가(요청자 기준). 비로그인 요청은 항상 false.

### Security 설정
- 공개(permitAll): 기존 전체 GET, `POST /auth/**`, 익명 후기 작성/삭제/도움돼요, `POST /images`.
- 인증 필요: `/auth/me`, `/me/**`.
- JWT 필터: Authorization 헤더 파싱 → 유효하면 SecurityContext 설정, 무효/부재면 **익명으로 통과** (401을 던지지 않음 — 공개 API가 많으므로). 보호 경로만 401.

## 3. UX 설계 (디자인 시안 톤 유지)

### 헤더 (AppLayout)
- 비로그인: 메뉴 끝에 `로그인` 텍스트 버튼(14/600, ink-mute, hover sea).
- 로그인: `로그인` 자리에 원형 아바타(28px, sea-soft 배경 + 닉네임 첫 글자 sea) → 클릭 시 드롭다운: 닉네임 표시 / "저장한 도감" / "로그아웃".
- "저장한 도감" 메뉴는 로그인 여부와 무관하게 유지 — 데이터 소스만 달라진다.

### 로그인/회원가입 (`/login`, `/signup` 페이지)
- 중앙 카드(max-w 400, rounded-card, border line): 로고 → 제목 → 필드 → sea 채움 버튼 → 하단 전환 링크.
- 로그인: "다시 오셨네요" / 이메일 / 비밀번호 / [로그인] / "처음이세요? 회원가입"
- 회원가입: "내 도감을 만들어보세요" + 부제 "저장한 생선과 후기를 어느 기기에서든" / 이메일 / 닉네임 / 비밀번호(헬퍼: "8자 이상이면 돼요") / [가입하기]
- 에러는 필드 아래 13px(기존 후기 폼 패턴 재사용). 로그인 실패는 폼 상단 한 줄.
- 로그인 성공 → 직전 페이지로 복귀(없으면 홈).

### localStorage 병합 (핵심 디테일)
- 로그인/가입 성공 시 localStorage 북마크가 1개 이상이면 모달 1회:
  "이 기기에 저장한 생선 N종이 있어요. 내 도감으로 옮길까요?" [옮기기] [나중에]
- [옮기기] → `POST /me/bookmarks/merge` → 성공 시 localStorage 클리어 + "내 도감으로 옮겼어요" 토스트.
- [나중에] → 그대로 두고 다시 묻지 않음(`bookmarkMergeDismissed` 플래그).

### 후기 폼 (로그인 모드)
- 로그인 상태: 닉네임·비밀번호 필드 숨김. 폼 상단에 "**회러버** 님으로 남깁니다" 캡션(13px ink-mute).
- 내 후기 카드: "삭제" 클릭 시 비밀번호 입력 없이 "이 후기를 지울까요?" 확인만.

### 지키는 것
- 비로그인 사용자에게 로그인 유도 배너·팝업을 띄우지 않는다. 로그인 진입점은 헤더 버튼 하나.
- 저장 하트는 비로그인도 그대로 동작(localStorage) — 하트 눌렀다고 로그인 창을 띄우지 않는다.

## 4. 티켓

### A1 — BE: User + 회원가입/로그인 + JWT
**프롬프트**:
> `docs/09_인증_내도감_설계.md` §1·§2 기준으로 인증을 구현해줘. `user` 도메인 패키지: User 엔티티, UserRepository, AuthController(`POST /auth/signup`, `POST /auth/login`, `GET /auth/me`), AuthService. spring-boot-starter-security + jjwt 의존성 추가. JWT는 HS256, `JWT_SECRET` 환경변수, 만료 7일. JwtAuthFilter는 토큰이 유효하면 SecurityContext에 userId를 넣고, 무효·부재면 익명으로 통과시켜(예외 던지지 말 것). SecurityConfig: §2의 공개/보호 경로 규칙, 기존 CORS 설정 유지, CSRF 비활성. 비밀번호는 기존 PasswordEncoder(BCrypt) 재사용. 검증 실패 400/중복 이메일 409/로그인 실패 401을 표준 에러 포맷으로.

**DoD**: 가입→로그인→`/auth/me` 흐름 동작, 잘못된 토큰으로 공개 API 호출 시 정상 응답(익명 취급), 기존 테스트 전부 통과.

### A2 — BE: 내 북마크 API
**프롬프트**:
> §2의 `/me/bookmarks` 4개 엔드포인트를 구현해줘. `user_bookmark` 복합키 엔티티(@EmbeddedId), GET은 저장 순 정렬로 `FishSummaryResponse[]` 반환(FishService의 집계 재사용 — N+1 금지, 기존 `findRatingStatsByFishIds` 활용). PUT/DELETE는 멱등(204). merge는 존재하는 fishId만 골라 일괄 insert(중복 무시). 전부 인증 필요, 미인증 401.

**DoD**: 추가/제거/목록/병합 동작, 미인증 401, 목록 응답에 avgRating 포함.

### A3 — BE: 후기 ↔ 회원 연결
**프롬프트**:
> §2 후기 변경사항을 구현해줘. Review에 nullable `user` @ManyToOne 추가. POST /fish/{id}/reviews: 인증된 요청이면 user 연결 + nickname은 회원 닉네임으로 강제, password는 무시(익명 요청은 기존 검증 그대로 — password 필수 유지). DELETE /reviews/{id}: 요청자가 후기 작성자(user 일치)면 password 없이 삭제 허용. ReviewResponse에 `mine` boolean 추가(비로그인 항상 false). 기존 익명 흐름 회귀 없어야 함.

**DoD**: 로그인 작성 시 user 연결·닉네임 자동, 내 후기 비번 없이 삭제, 익명 흐름 기존과 동일, 테스트 통과.

### A4 — FE: 인증 상태·토큰 관리
**프롬프트**:
> FE에 인증 기반을 만들어줘. `src/api/auth.ts`(signup/login/me), `src/hooks/useAuth.ts` — 토큰은 localStorage `fishnote:accessToken`, 사용자 정보는 React Query `['auth','me']`로 관리(토큰 없으면 비활성). axios 인터셉터: 토큰 있으면 Authorization 헤더 자동 첨부, 401 응답 시 토큰 제거. logout()은 토큰·캐시 클리어. 라우트 `/login`, `/signup` 골격 추가.

**DoD**: 토큰 저장/첨부/만료 처리 동작, 새로고침 후 로그인 유지.

### A5 — FE: 로그인/회원가입 화면
**프롬프트**:
> §3의 UX대로 `/login`·`/signup` 페이지를 구현해줘. 중앙 카드(max-w 400), 기존 후기 폼의 Field·inputClass 패턴 재사용, 카피는 §3 그대로. 성공 시 직전 페이지 복귀(react-router state 활용, 없으면 홈). 로그인 실패는 폼 상단 "이메일 또는 비밀번호를 확인해 주세요". 헤더(AppLayout): 비로그인 '로그인' 버튼 / 로그인 시 아바타+드롭다운(저장한 도감·로그아웃).

**DoD**: 가입→자동 로그인→헤더 아바타 표시, 로그아웃 동작, 모바일 폭 정상.

### A6 — FE: 북마크 서버 연동 + 병합
**프롬프트**:
> `useBookmarks`를 확장해줘: 로그인 상태면 서버(`/me/bookmarks`) 기반(React Query + PUT/DELETE mutation, 낙관적 업데이트), 비로그인이면 기존 localStorage 그대로. 컴포넌트 쪽 인터페이스(isBookmarked/toggleBookmark/bookmarkCount)는 변경 없이 내부만 분기. SavedPage도 로그인 시 서버 목록 사용. 로그인·가입 직후 localStorage 북마크가 있으면 §3의 병합 모달 1회 노출 → merge API → 성공 시 localStorage 클리어.

**DoD**: 로그인 상태 북마크가 서버에 저장(다른 브라우저에서 로그인해도 유지), 병합 모달 1회 동작, 비로그인 흐름 기존과 동일.

### A7 — FE: 후기 폼 로그인 모드
**프롬프트**:
> ReviewForm: 로그인 상태면 닉네임·비밀번호 필드를 숨기고 상단에 "{닉네임} 님으로 남깁니다" 캡션, 제출 시 두 필드 없이 전송. ReviewList: `mine`인 후기는 삭제 클릭 시 비밀번호 입력 없이 확인 문구만("이 후기를 지울까요?") 후 DELETE(비번 없이). 익명 후기 삭제 흐름은 기존 유지.

**DoD**: 로그인 작성 시 2필드 미노출, 내 후기 원클릭 삭제, 익명 흐름 회귀 없음.

### A8 — 검수
- 가입 → 로그인 → 하트 저장 → 다른 브라우저 로그인 → 저장 유지 확인.
- 익명(시크릿 창): 후기 작성·삭제·저장 전부 기존처럼 동작.
- 병합: localStorage에 2종 저장 → 로그인 → 모달 → 옮기기 → 저장한 도감 확인.
- JWT_SECRET 미설정 시 부팅 실패 여부 확인(환경변수 필수 — Render에 먼저 추가!).
- 모바일 폭에서 로그인/드롭다운 확인.

## 5. 배포 준비물
- Render 환경변수 `JWT_SECRET` 추가 (32자 이상 무작위 문자열 — `openssl rand -base64 48`로 생성). **A1 푸시 전에 먼저.**
- DB 변경은 Flyway 버전 마이그레이션으로 적용하고, Hibernate는 `ddl-auto: validate`로 매핑만 검증한다. `review.user_id`는 nullable이라 기존 데이터 안전.

## 6. v2 확장 메모 (이번엔 안 함)
- refresh 토큰 + 자동 갱신, 이메일 인증, 비밀번호 재설정, 관리자 롤(role 컬럼) — 관리자 페이지는 이 기반 위에 올린다.

## 7. 카카오 OAuth 구현

- FE가 카카오 인가 엔드포인트로 이동하고 `/auth/kakao/callback`에서 `code`와 CSRF 방지용 `state`를 확인한다.
- FE는 `POST /api/v1/auth/kakao`로 `code`, `redirectUri`를 전달한다. BE만 Client Secret을 사용해 카카오 토큰과 사용자 정보를 조회한다.
- 카카오 토큰은 저장하지 않고, 확인이 끝나면 기존 FishNote JWT를 발급해 이메일 로그인과 같은 인증·북마크 흐름을 사용한다.
- 검증된 카카오계정 이메일이 제공되면 같은 이메일의 기존 회원에 로그인 수단을 연결한다. 이메일 권한이 없으면 카카오 서비스 사용자 식별자로 전용 계정을 만든다.
- 연결 정보는 `user_oauth_account`에 저장하며, 카카오 전용 회원은 비밀번호 없이 계정 삭제 확인 문구만으로 탈퇴할 수 있다.
